# リファクタリングワークフロー

リファクタリングを求められた場合、以下の厳密な順序に従うこと:

## フェーズ 1: テスト検証（リファクタリング前）

1. **既存テストの確認**
   - リファクタリング対象コードをカバーするテストを特定する
   - 関連テストを実行し、現在の動作を確認する

2. **不足テストの作成（必要な場合）**
   - リファクタリング範囲をカバーするテストがない場合、先にテストを書く
   - テストはリファクタリング後も維持されるべき期待動作を検証すること
   - 新しいテストが現在の実装でパスすることを確認する

3. **テストを先にコミット**
   - `/git` コマンドでテスト変更をコミットする
   - `.claude/commands/git.md` の全ルールに従うこと
   - コミットメッセージ例: `test(package): add tests for X before refactoring`

## フェーズ 2: リファクタリング実装（テストコミット後）

4. **リファクタリング開始**
   - テストがコミットされた後にのみリファクタリングを開始する
   - 段階的に変更する
   - 動作が維持されていることを確認するため、頻繁にテストを実行する

5. **リファクタリング検証**
   - 関連テストを全て実行する: `yarn test`
   - ビルドチェックを実行する: `yarn build`
   - lint を実行する: `yarn lint`

6. **リファクタリングをコミット**
   - `/git` コマンドでリファクタリング変更をコミットする
   - コミットメッセージ例: `refactor(package): simplify X implementation`

## 基本原則

- **テストなしでリファクタリングしない**: テストが安全ネット
- **テストが先、リファクタリングが後**: 必ずテストをコミットしてからリファクタリング
- **動作を維持する**: リファクタリングは外部動作を変えてはならない
- **小さな増分**: 小さく検証可能な変更を行う
